Today was a turning point — I finally got Aegis generating text correctly after a week of fighting broken code.

## The Diagnosis

I did a full code audit of `main.cpp` and found the root cause of my problems. The biggest bug was on this line:

```cpp
llama_batch next_batch = llama_batch_get_one(&next_token, 1);
next_batch.pos[0] = n_past;  // ← THIS IS UNDEFINED BEHAVIOR
```

`llama_batch_get_one()` returns a batch where `pos` is `NULL`. That's intentional — `NULL` is the signal to llama.cpp to auto-track positions internally (it just counts up: 0, 1, 2, 3...). When I wrote `next_batch.pos[0] = n_past`, I was dereferencing a null pointer. The fact it didn't crash immediately was pure luck — it was writing to whatever memory address happened to be there.

The fix was dead simple: delete that line entirely. Let auto-tracking do its job.

## Other Fixes Applied

- **No GPU offloading**: `model_params.n_gpu_layers` defaulted to `0`, meaning all 24 layers ran on CPU. Added `model_params.n_gpu_layers = -1` to push everything to my RTX 3050. Inference went from painfully slow to instant.
- **Double `llama_sampler_accept()`**: `llama_sampler_sample()` already calls accept internally (it's in the docs at llama.h:1462). I was calling it again manually, which double-notifies penalty samplers. Removed both redundant calls.
- **No tokenization error check**: `llama_tokenize()` returns negative on failure. I wasn't checking. Added the guard.
- **Buffer too small**: `buf[64]` for token-to-text conversion. Some tokens (Unicode, multi-byte) can exceed that. Bumped to `buf[256]`.

## The Sampling Seed Discovery

After fixing the bugs, I switched from greedy sampling to `temp(0.8) + dist(42)` for creative output. But the output was identical every run. Spent 10 minutes confused before realizing: `42` is a **fixed seed**. Same seed → same random number sequence → same output. Changed to `LLAMA_DEFAULT_SEED` (0xFFFFFFFF) which tells llama.cpp to pick a random seed, and now I get different outputs each run as expected.

## Where I Actually Stand

The honest truth: my `main.cpp` is a working "hello world" for llama.cpp inference. It loads a model, tokenizes a prompt, runs the forward pass, and prints generated text token by token. But it has zero KV cache eviction logic — which is the entire point of Aegis. The real work starts now: reading the KV cache source code (`llama-kv-cache.h/cpp`), understanding cell tracking, and figuring out where to inject my S_i scoring formula into the inference pipeline.
