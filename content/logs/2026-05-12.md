other than this i have also read paper on H2O caching, where i learned 4 diffrent ways which they compared at the start which were pretty basic in terms of what they had implemented. 
1.) Dynamic sparcity - computationally expensive
2.) static sparcity(strided) - Keeps a sliding window of recent tokens (the diagonal line) plus every $N$-th token from the past (the periodic vertical stripes).
3.) static sparcity(static) - Keeps only a fixed window of the most recent tokens. Everything older is trashed.
4.) Static Sparsity w. $H_2O$ (Heavy Hitters) - H2O keeps two sets of tokens:
1. Recent tokens    — last W tokens always kept (local window)
2. Heavy hitters    — tokens with highest ACCUMULATED attention score
example : if token "Paris" received attention 0.4 from token 10, 0.3 from token 11, 0.5 from token 12 — its accumulated score = 1.2.
Token "the" received 0.02 from token 10, 0.01 from token 11, 0.02 from token 12 — accumulated score = 0.05.
H2O keeps "Paris", evicts "the".

on top of just talking about the storing of KV-cache problem they also pointed out to problem for memory needed to compute attention, which is solved by a hardware hack known as flash attention as explained below:
GPU has 2 parts i) HBM (High Bandwidth Memory) - The main memory, huge and slow. and ii)SRAM (Static RAM) - The memory right next to the compute cores, fast and small. all math happens in SRAM. and in flash attention we Load a small "tile" (block) of $Q, K,$ and $V$ from HBM into the fast SRAM. Calculates the attention scores for just that block.Calculates the softmax on the fly in SRAM (using a math trick called "online softmax"). Writes the final result directly back to HBM.

though it is not at all related to the main problem of kv cahing, it was pretty interesting to learn about.

what is a good KV-cache eviction policy? the given is a trade-off triangle
--> to have a production system we need 
1.) Small Cache Size (Memory) - This is the most important, that we cant store large amount of K, V vectors in memory
2.) Low Miss Rate (Accuracy) - we must keep correct tokens, giving highest attention scores.
3.) Low-Cost Eviction (Speed) - the algorithm must be fast

3 challanges - First, it is not immediately clear whether the size of the KV cache can be restricted—each decoding step might, in principle, require access to all previous attention keys and values. Second, identifying an optimal eviction policy that maintains generation accuracy is a combinatorial problem. Finally, even if an optimal policy can be brute-forced, it is infeasible for deployment on real-world applications.
